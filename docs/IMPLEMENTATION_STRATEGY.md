## 1. 기술 구현 전략 (밭 자동설계)

### 1.1 알고리즘 단순화 전략

**핵심 인사이트**: 원래 계획의 5개 알고리즘(Bin Packing, Grid Layout, A*, Flow Network, Voronoi)은 과잉. **MVP는 2개면 충분**.

#### MVP 알고리즘: Grid + Polygon Clipping

```
필요 알고리즘:
  1. Grid Layout (나무 배치) → Turf.js pointGrid()
  2. Polygon Clipping (경계 처리) → Turf.js pointsWithinPolygon()

이 2개로 MVP의 90% 구현 가능.
나머지는 Phase 2~3에서 점진 추가.
```

#### Turf.js 핵심 함수 (프론트엔드만으로 동작)

```javascript
// 1. 나무 배치 (핵심 로직 - 이것만으로 MVP 동작)
import { pointGrid, pointsWithinPolygon, area, buffer } from '@turf/turf';

function designOrchard(farmPolygon, rowSpacing, treeSpacing) {
  // 밭 경계의 바운딩박스 내에 그리드 포인트 생성
  const bbox = turf.bbox(farmPolygon);
  const cellSide = Math.min(rowSpacing, treeSpacing) / 1000; // km 단위
  const grid = turf.pointGrid(bbox, cellSide, { units: 'kilometers' });

  // 밭 경계 안에 있는 포인트만 필터링
  const treesInFarm = turf.pointsWithinPolygon(grid, farmPolygon);

  // 경계에서 buffer(여백) 적용 (밭 가장자리 제외)
  const innerFarm = turf.buffer(farmPolygon, -2, { units: 'meters' }); // 2m 여백
  const finalTrees = turf.pointsWithinPolygon(treesInFarm, innerFarm);

  return finalTrees;
}

// 2. 동선 설계 (Phase 2)
// 입구 위치에서 가장 가까운 열부터 S자 패턴 생성
// A* 대신 단순 S-패턴으로 충분

// 3. 관수 경로 (Phase 3)
// Flow Network 대신 열 따라 직선 배관 → 지관 분기
// 실제 농가에서도 이 방식이 표준
```

#### 좌표계 변환 (필수)

```javascript
// 한국 좌표계(EPSG:5179/5174) → WGS84(EPSG:4326) 변환
import proj4 from 'proj4';

// 한국 좌표계 정의
proj4.defs('EPSG:5179', '+proj=tmerc +lat_0=38 +lon_0=127.5 +k=0.9996 +x_0=1000000 +y_0=2000000 +ellps=GRS80 +units=m +no_defs');

function convertToWGS84(x, y) {
  return proj4('EPSG:5179', 'EPSG:4326', [x, y]);
}
```

### 1.2 3D 시각화 전략

### 1.3 백엔드 선택: Python FastAPI 권장

**결론**: Node.js보다 **Python FastAPI** 권장

| 기준 | Node.js | Python FastAPI | 판정 |
|------|---------|----------------|------|
| API 연동 (기상청, KAMIS) | 가능 | **공공 API 예제가 Python 위주** | Python |
| GIS/지리 연산 | Turf.js (프론트) | Shapely, GeoPandas | **둘 다 가능** |
| ML 모델 (병해충 예측) | ONNX Runtime | **scikit-learn, PyTorch 네이티브** | Python |
| 데이터 처리 | 가능 | **Pandas, NumPy 생태계** | Python |
| 성능 | 높음 | **FastAPI = Node.js급 성능** | 동급 |
| 기존 경험 (PJ06) | - | - | 확인 필요 |

```
추천 아키텍처:
  프론트엔드: Next.js + Turf.js (지도/설계 연산)
  백엔드: Python FastAPI (API 연동, ML, 데이터 처리)
  DB: PostgreSQL + PostGIS
  캐시: Redis (경매가, 기상 데이터)
```

### 3.1 사업 계획서 PDF 자동 생성 (최고 가치)

**노력**: 낮음 (1~2주) | **효과**: 매우 높음

```
기능 설명:
  밭 설계 결과 + 수익 시뮬레이션 → 자동 PDF 생성

PDF 내용:
  1. 토지 개요 (위치, 면적, 지가)
  2. 설계도 (2D 배치도 이미지)
  3. 투자 비용 상세 내역
  4. 연차별 수익 시뮬레이션 그래프
  5. 손익분기점 분석
  6. 품종 비교 분석

활용처:
  - 귀농 창업 사업 계획서
  - 농협 대출 심사 서류
  - 정부 보조금 신청 첨부 자료
  - 투자자 설명 자료

기술:
  - React-PDF 또는 jsPDF + html2canvas
  - 서버 사이드: Puppeteer로 HTML → PDF
  - 디자인: Mirai 캐릭터 워터마크 + 브랜딩

💰 수익화: 무료 1회, 이후 건당 5,000원 또는 프리미엄 구독
```

**왜 최고 가치인가?**
- 밭 설계는 1회성이지만 PDF는 보관/공유됨
- 대출 심사에 필요 → 실질적 금전 가치
- 자연스러운 바이럴 (농협 담당자에게 공유)
- 경쟁사 없음 (이런 서류를 자동 생성하는 곳이 없음)

### 3.4 농업기술센터 협업 모델

**노력**: 영업 활동 | **효과**: 매우 높음

```
한국 전국 시군구에 농업기술센터 존재 (약 160개)
사과 주산지 농업기술센터 타겟 (약 20~30개):
  - 영주, 안동, 청송, 의성, 영양 (경북)
  - 충주, 제천 (충북)
  - 거창, 함양 (경남)
  - 양구, 정선 (강원)
  - 장수, 무주 (전북)

협업 모델:
  1. 무료 제공: 농업기술센터에 플랫폼 무료 사용 제공
  2. 교육 연계: 농업인 교육 시 PJ18_APPLE 데모
  3. 데이터 교환: 지역 재배 데이터 ↔ 플랫폼 분석 리포트
  4. 전문가 연결: 센터 컨설턴트를 플랫폼 전문가로 등록
  5. 공동 연구: R&D 과제 공동 참여

가치:
  - 사용자 확보 채널 (센터가 농업인에게 추천)
  - 데이터 확보 (현장 데이터)
  - 신뢰도 확보 (공공기관 인증)
  - 정부 사업 참여 기반
```

### 3.5 기후 변화 재배적지 이동 지도

**노력**: 중간 (2~3주) | **효과**: 높음

```
기능:
  한국 지도 위에 사과 재배 적합 지역을 연도별 애니메이션으로 표시

데이터:
  - 기상청 기후 변화 시나리오 데이터 (공개)
  - 사과 재배 적합 조건 (유효적산온도 1,200~1,700℃, 저온요구도 800~1,500시간)
  - IPCC SSP 시나리오별 예측

시각화:
  - Leaflet 히트맵 레이어
  - 슬라이더로 연도 조절 (2026→2050)
  - 색상 그라데이션 (적합→부적합)

가치:
  - 콘텐츠 마케팅 핵심 자료 (유튜브 영상 소재)
  - 장기 투자 의사결정 (어디에 과수원을 만들까)
  - 정부/연구기관 관심 유도
  - SNS 바이럴 가능성 높음
```

### 3.6 연동 추천: 흙토람(토양정보) + 밭 설계

```
연동 효과:
  현재: 토지 선택 → 면적/지가만 표시
  강화: 토지 선택 → 면적 + 지가 + 토양 유형 + 적합도 점수

  "이 토지는 양토(壤土)로, 사과 재배 적합도 A등급입니다.
   배수가 양호하며 M.9 대목에 적합합니다.
   추천 비료: 퇴비 2톤/10a + 석회 200kg/10a"

흙토람 API 존재 시:
  → 위치 기반 토양 정보 자동 조회
  → 밭 설계 입력 데이터 품질 대폭 향상

흙토람 API 부재 시:
  → 흙토람 웹사이트에서 주요 사과 산지 토양 데이터 수작업 수집
  → JSON 파일로 저장 → 위치 기반 매칭
```

### 4.3 "치트 코드" (노력 거의 없이 가치 추가)

```

2. 경매가 알림 (가격 상승/하락 시)

3. 품종 비교표 자동 생성

   → 비교 분석 가능

5. 공공데이터포털 오픈소스 코드 활용
   → GitHub에 "기상청 API", "KAMIS API" 검색
   → 이미 구현된 래퍼 라이브러리 다수 존재
   → 바퀴 재발명 방지
```

## 6. 1인 개발 생존 전략

### 6.1 시간 배분 (주당 40시간 기준)

```
Phase 1 (4주):
  코딩: 70% (28시간/주)
  API 테스트/디버깅: 15% (6시간/주)
  기획/디자인: 10% (4시간/주)
  농업인 인터뷰/리서치: 5% (2시간/주)
```

### 6.2 자동화 우선

```
CI/CD:
  - Vercel 자동 배포 (프론트엔드)
  - Railway 또는 Render 자동 배포 (백엔드)
  - GitHub Actions (린트, 빌드 검증)

데이터 수집:
  - cron job (기상: 매시간, 경매가: 일 2회, 통계: 월 1회)
  - 장애 알림 (Slack/Discord webhook)

모니터링:
  - Vercel Analytics (무료)
  - Sentry (에러 추적, 무료 tier)
  - 간단한 헬스체크 API (/api/health)
```

### 6.3 기술 부채 관리

```
Phase 1 허용하는 부채:
  ✅ 하드코딩된 품종 데이터 (JSON 파일)
  ✅ 단순 인메모리 캐시 (Redis 없이)
  ✅ 2D 지도만 (3D 없이)
  ✅ 카카오 로그인만 (다른 소셜 없이)
  ✅ 한국어만 (다국어 없이)

Phase 2에서 해결:
  → 품종 DB 마이그레이션
  → Redis 도입
  → 인증 확장
```

### 6.4 외부 리소스 활용

```
무료/저비용 도구:
  - Vercel: 프론트 호스팅 (무료 tier 충분)
  - Railway/Render: 백엔드 호스팅 ($5~20/월)
  - Neon/Supabase: PostgreSQL (무료 tier)
  - Upstash: Redis (무료 tier, 10K 요청/일)
  - Cloudflare R2: 파일 저장 (무료 10GB)
  - Resend: 이메일 발송 (무료 100건/일)

AI/ML:
  - HuggingFace: 사전학습 모델 (무료)
  - Google Colab: 모델 학습 (무료 GPU)
  - OpenAI/Claude API: Mirai 챗봇 ($10~50/월)

디자인:
  - shadcn/ui: UI 컴포넌트 (무료)
  - Heroicons/Lucide: 아이콘 (무료)
  - Figma: UI 설계 (무료 tier)

총 예상 초기 비용: 월 $20~80 (약 3~10만원)
```
